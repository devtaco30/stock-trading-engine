# 🎯 NPL 프로젝트 - 백엔드 딥다이브 가치 평가

## 1. 이 프로젝트에서 다룰 수 있는 기술 주제

### 1-1. 동시성 제어
**상황:**
- 49명 정원에 10만 명 동시 접근
- 상품 100개 동시 오픈 시 독립적 락 관리
- 분산 환경에서 정확히 49명만 받기

**기술:**
- Redis 분산 락
- Optimistic Lock (JPA @Version)
- Pessimistic Lock (SELECT FOR UPDATE)
- Lua Script (원자성)

### 1-2. 트랜잭션 관리
**상황:**
- 투자 신청 → 입금 → 확정 → 경매 → 배당까지의 긴 흐름
- 중간에 실패 시 롤백 또는 보상 처리

**기술:**
- Saga Pattern
- Event Sourcing
- 보상 트랜잭션
- 상태 머신

### 1-3. 대량 데이터 처리
**상황:**
- 수천 명에게 동시 배당 정산
- BigDecimal 정밀 계산
- 배치 성능 최적화

**기술:**
- Spring Batch
- 청크 처리
- 병렬 처리
- 멱등성 보장

### 1-4. 실시간 처리
**상황 (유통 시장 포함 시):**
- 유통 시장의 실시간 호가
- 매수/매도 매칭 엔진
- 실시간 알림

**기술:**
- WebSocket
- SSE (Server-Sent Events)
- 인메모리 데이터 구조
- 이벤트 스트리밍

### 1-5. 금융 데이터 정합성
**상황:**
- 돈이 오가는 모든 과정의 정확성
- 1원 단위까지 정확한 정산
- 감사 추적 (Audit Trail)

**기술:**
- BigDecimal 정밀 연산
- 이중 장부 (Double-entry)
- 이력 관리 (Auditing)
- 불변 이벤트 저장

## 2. 다른 프로젝트와 비교

### 2-1. 티켓 예매 시스템 (콘서트, 항공권)

**유사점:**
- ✅ 선착순 경쟁
- ✅ 제한된 좌석
- ✅ 높은 동시성

**차이점:**
- ❌ 티켓: 결제 후 바로 끝
- ✅ NPL: 결제 → 6개월 대기 → 배당 (긴 라이프사이클)
- ✅ NPL: 정밀한 금액 계산 필요 (BigDecimal)
- ✅ NPL: 지분 개념 (소수점 지분율)

**평가:**
- NPL이 트랜잭션 복잡도는 높음
- 티켓이 순간 동시성은 더 높음

### 2-2. 주식 거래소

**유사점:**
- ✅ 실시간 매칭 (유통 시장 포함 시)
- ✅ 호가 관리
- ✅ 높은 TPS

**차이점:**
- ❌ 주식: 초당 수만 건 거래
- ✅ NPL: 초기 청약 시에만 트래픽 집중
- ✅ NPL: 배당 정산 로직 추가
- ✅ NPL: 긴 라이프사이클 관리

**평가:**
- 유통 시장 포함 시 주식 거래소와 유사
- NPL이 도메인 복잡도는 높음

### 2-3. P2P 대출 플랫폼

**유사점:**
- ✅ 투자자 모집
- ✅ 원금+이자 정산
- ✅ 금융 규제 준수

**차이점:**
- ❌ P2P: 대출자 직접 연결
- ✅ NPL: 경매 낙찰 프로세스 포함
- ✅ NPL: 채권자 지위, 우선권 개념
- ✅ NPL: 선착순 경쟁 (P2P는 여유 있음)

**평가:**
- NPL이 동시성 이슈는 높음
- P2P가 신용 평가 로직 복잡

## 3. 딥다이브 가치 평가

### 3-1. 충분히 가치 있는 경우

**✅ Option A: 유통 시장(거래소) 포함**
- 실시간 매칭 엔진 구현
- 호가 관리, 체결 로직
- 주식 거래소급 난이도
- 포트폴리오 임팩트: ⭐⭐⭐⭐⭐

**✅ Option B: 상품 100개 동시 오픈**
- 독립적 동시성 제어
- 상품별 락 관리
- 대규모 트래픽 처리
- 포트폴리오 임팩트: ⭐⭐⭐⭐

**✅ Option C: 복잡한 정산 로직**
- 배당, 수수료, 이자 계산
- BigDecimal 정밀 연산
- Spring Batch 최적화
- 포트폴리오 임팩트: ⭐⭐⭐⭐

**✅ Option D: 긴 라이프사이클**
- 상태 머신 설계
- Saga Pattern
- Event Sourcing
- 포트폴리오 임팩트: ⭐⭐⭐⭐⭐

### 3-2. 가치가 부족한 경우

**❌ 청약만 하고 끝**
- 단순 예매 시스템과 비슷
- 배당 정산 없음 → 금융 시스템의 핵심 빠짐
- 2주면 끝남

**❌ 상품 1개씩만 오픈**
- 트래픽 분산되어 난이도 낮음
- 동시성 제어 연습 부족

**❌ 49명 제한만 그대로**
- "겨우 49명" 문제
- 확장성 없음

## 4. 프로젝트 스케일 결정

### 4-1. Option 1: 미니멀 (❌ 딥다이브 부족)

**기능:**
- 청약만 가능
- 상품 1개씩 오픈
- 배당 정산 없음
- 49명 제한

**딥다이브 주제:**
- 동시성 제어 (1개만)

**기간:** 2주

**평가:**
- ❌ 배울 게 별로 없음
- ❌ 포트폴리오 임팩트 낮음
- ❌ 시간 낭비 가능성

### 4-2. Option 2: 스탠다드 (✅ 적당함)

**기능:**
- 청약 + 배당 정산
- 상품 10~20개 동시 오픈
- 49명 또는 무제한(공모)
- 대기자 관리
- 취소/환불 처리

**딥다이브 주제:**
- 동시성 제어 (복수 상품)
- 정산 로직 (Batch)
- 트랜잭션 관리
- 상태 머신

**기간:** 1~2개월

**평가:**
- ✅ 충분히 배울 게 있음
- ✅ 포트폴리오 임팩트 높음
- ✅ 완성 가능한 범위

### 4-3. Option 3: 풀스택 (⚠️ 거래소급, 범위 큼)

**기능:**
- 청약 + 배당 + 유통 시장
- 상품 100개 동시 오픈
- 실시간 지분 거래
- 복잡한 정산 (수수료, 세금 등)
- 실시간 호가창

**딥다이브 주제:**
- 동시성 제어 (대규모)
- 실시간 매칭 엔진
- 정산 로직 (복잡)
- Event Sourcing
- 고성능 최적화
- WebSocket

**기간:** 3~6개월

**평가:**
- ✅ 포트폴리오급
- ⚠️ 범위가 너무 큼
- ⚠️ 완성 못 할 수도
- ✅ 최고 난이도

## 5. 기술 스택 검증

### 5-1. 확실히 써볼 수 있는 것

**✅ Redis 분산 락:**
- 49명 제한 관리에 필수
- Redisson 사용
- 타임아웃, 재시도 로직

**✅ Spring Batch:**
- 배당 정산에 필수
- 청크 처리
- 멱등성 보장

**✅ Saga Pattern:**
- 긴 트랜잭션 관리
- 보상 트랜잭션
- 상태 머신

**✅ Event Sourcing (일부):**
- 투자 이력 관리
- Audit Trail
- 과거 시점 재현

**✅ BigDecimal:**
- 금액 정밀 계산
- 소수점 지분율

**✅ JPA Pessimistic Lock:**
- 동시성 제어
- SELECT FOR UPDATE

### 5-2. 애매한 것

**⚠️ Java 21 Virtual Threads:**
- 배당 정산 시 병렬 처리에 사용?
- 일반 Thread Pool로 충분할 수도
- 학습 가치는 있음

**⚠️ Sealed Interface:**
- 자산 타입 추상화에 쓴다는데?
- 굳이 필요한가?
- enum으로 충분할 수도

### 5-3. 과한 것

**❌ LMAX Disruptor:**
- 유통 시장 없으면 불필요
- 초당 수만 건 처리는 현실적으로 없음

**❌ Kafka:**
- 이벤트 스트리밍까지 필요한가?
- 규모에 비해 과함

- "왜 이 프로젝트를 했나?" 답변 가능

**⚠️ 유통 시장 없으면:**
- 단순 예매 시스템과 비슷

**⚠️ NPL 도메인 지식:**

## 7. 대안 프로젝트 아이디어

### 7-1. 만약 NPL이 부족하다면?

**1. 암호화폐 거래소 (모의)**
- 실시간 호가 매칭
- 초당 수만 건 주문 처리
- WebSocket 실시간 업데이트
- 정밀한 잔고 관리

**2. 티켓팅 시스템 (콘서트, 스포츠)**
- 초고속 선착순 처리
- 대기열 관리
- 결제 타임아웃 처리
- 재판매 시장

**3. 실시간 경매 플랫폼**
- 실시간 입찰
- 최고가 갱신
- 마감 임박 알림
- 즉시 낙찰 처리

**4. 배달 주문 플랫폼**
- 실시간 주문 매칭
- 배달 기사 할당
- 동시 주문 처리
- 실시간 위치 추적
