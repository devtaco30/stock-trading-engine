# 주식 거래 엔진 - 4단계 시나리오

## 1단계: 기본 주문 및 체결

**시나리오:**
계좌 A가 삼성전자 100주 매수.

**처리 과정:**

### 주문 시점 (09:00)
- 삼성전자 현재가: 70,000원
- 지정가 주문: 70,500원에 100주 매수
- 필요 금액: 7,050,000원
- 증거금 40%: 2,820,000원

### 매수 가능 금액 체크
```
현금 잔액: 3,000,000원
매수 가능 금액 = 3,000,000 / 0.4 = 7,500,000원
7,050,000원 < 7,500,000원 → 주문 가능
```

### 증거금 홀딩
- 현금에서 2,820,000원 차감
- 잔여 현금: 180,000원
- 주문 상태: 대기

### 체결 (09:15)
- 삼성전자 가격 상승: 70,500원
- 체결 완료
- 보유 주식에 삼성전자 100주 추가
- 미결제 금액: 4,230,000원 (수요일 납부 예정)

**현재 상태:**
```
현금: 180,000원
보유 주식: 삼성전자 100주 (매수가 70,500원, 현재가 70,500원)
미결제: 4,230,000원
```

## 2단계: T+2 시차 활용 (회전매매)

**시나리오:**
같은 날 오후, 계좌 A가 삼성전자 매도하고 카카오 매수.

### 삼성전자 매도 (14:00)
- 삼성전자 현재가: 72,000원 (상승)
- 100주 전량 매도 (시장가)
- 체결가: 72,000원
- 매도 대금: 7,200,000원
- 실제 입금일: 수요일 (T+2)

### 매도 예정금액 발생
- 수요일에 들어올 돈: 7,200,000원
- 현재는 아직 입금 안 됨

### 카카오 매수 시도 (14:30)
- 카카오 현재가: 50,000원
- 목표: 200주 매수
- 필요 금액: 10,000,000원
- 증거금 40%: 4,000,000원

### 매수 가능 금액 재계산
```
현금: 180,000원
매도 예정금액: 7,200,000원
합계: 7,380,000원

매수 가능 금액 = 7,380,000 / 0.4 = 18,450,000원
```

카카오 10,000,000원 매수 가능.

### 카카오 체결 (14:31)
- 카카오 가격 변동: 50,200원 (약간 상승)
- 200주 체결
- 실제 매수 금액: 10,040,000원
- 증거금 40%: 4,016,000원 차감

**현재 상태:**
```
현금: 180,000원 (변동 없음, 매도 예정금액에서 차감)
보유 주식: 카카오 200주 (매수가 50,200원)
매도 예정금액: 3,184,000원 (7,200,000 - 4,016,000)
미결제:
  - 삼성전자 매수분: 4,230,000원
  - 카카오 매수분: 6,024,000원
  총: 10,254,000원
```

### 화요일 상황
카카오 가격 하락: 48,000원

보유 주식 평가액 = 200주 x 48,000원 = 9,600,000원

### 수요일 결제일
```
[입금]
삼성전자 매도 대금: 7,200,000원

[출금]
삼성전자 매수 잔금: 4,230,000원
카카오 매수 잔금: 6,024,000원
합계: 10,254,000원

[정산]
현금: 180,000원
입금: 7,200,000원
출금: 10,254,000원
최종: 180,000 + 7,200,000 - 10,254,000 = -2,874,000원

→ 미수금 2,874,000원 발생
```

## 4단계: 강제 청산 (반대매매)

**시나리오:**
목요일까지 입금 없음. 금요일 장 시작 전 강제 청산.

### 목요일 종가
카카오: 47,500원 (계속 하락)

### 금요일 새벽 05:00 - 청산 준비

**미수 계좌 조회:**
```sql
SELECT * FROM account 
WHERE status = 'IN_ARREARS' 
AND debt_days >= 1;
```

**청산 수량 계산:**
```
미수금: 2,874,000원
보유 주식: 카카오 200주
전일 종가: 47,500원
하한가 (안전 마진): 42,750원 (전일 종가의 -10%)

청산 필요 수량 = 2,874,000 / 42,750 = 67.2주 → 68주
```

하한가 기준으로 계산하는 이유: 시초가가 급락할 수 있어서 안전 마진 확보.

**강제 매도 주문 생성:**
```sql
INSERT INTO order (
  account_id, stock_code, quantity, 
  price, type, created_at
) VALUES (
  'ACC001', 'KAKAO', 68, 
  'MARKET_PRICE', 'FORCED_SELL', NOW()
);
```

### 금요일 장 시작 (09:00) - 시초가 체결

카카오 시초가: 46,800원 (예상보다 높게 형성)

```
체결 수량: 68주
체결가: 46,800원
체결 금액: 3,182,400원
```

**미수금 청산:**
```sql
UPDATE debt 
SET amount = amount - 3182400, status = 'SETTLED'
WHERE account_id = 'ACC001';
```

**계좌 정상화:**
```sql
UPDATE account
SET status = 'ACTIVE', frozen_at = NULL
WHERE id = 'ACC001';
```

**최종 상태:**
```
현금: 308,400원 (3,182,400 - 2,874,000)
보유 주식: 카카오 132주 (200 - 68)
  평가액: 132주 x 46,800원 = 6,177,600원
미수금: 0원
상태: 정상
```

**실제 손실:**
```
초기 현금: 3,000,000원
최종 자산: 308,400원 (현금) + 6,177,600원 (주식) = 6,486,000원
손실: 3,000,000 - 6,486,000 = -3,486,000원 (손실 아님, 수익)

아 잠깐, 계산 다시:
- 삼성전자 100주 매수: 7,050,000원
- 삼성전자 100주 매도: 7,200,000원 (+150,000원)
- 카카오 200주 매수: 10,040,000원
- 카카오 68주 강제 매도: 3,182,400원
- 카카오 132주 보유: 6,177,600원

총 투입: 7,050,000원
총 회수: 7,200,000 + 3,182,400 + 6,177,600 = 16,560,000원

어? 이상한데... 다시 계산

초기 현금: 3,000,000원만 있었음
최종: 308,400 + 6,177,600 = 6,486,000원
수익: +3,486,000원

맞다, 레버리지 효과로 수익 난 거임.
하지만 미수금 때문에 강제 청산당해서 원하는 타이밍에 못 팔았음.
```

## 가격 변동이 미치는 영향

### 1. 매수 가능 금액 변동

보유 주식의 대용가는 현재가 기준으로 계속 변동.

```java
// 14:00 시점
카카오 현재가: 50,000원
대용가 = 200주 x 50,000 x 0.7 = 7,000,000원

// 14:30 시점 (가격 하락)
카카오 현재가: 48,000원
대용가 = 200주 x 48,000 x 0.7 = 6,720,000원

// 매수 가능 금액 280,000원 감소
```

실시간으로 매수 가능 금액이 변한다.

### 2. 강제 청산 수량 변동

청산 수량 계산 시 현재가 사용.

```
전일 종가 47,500원 기준 → 68주
장중 45,000원으로 하락 시 → 73주 필요

가격 하락하면 더 많이 팔아야 함.
```

### 3. 체결가와 예상가 차이

지정가 주문: 원하는 가격에 안 될 수도 있음
시장가 주문: 예상과 다른 가격에 체결될 수 있음

강제 청산은 시장가 매도라 시초가에 따라 체결가 변동.

## 구현 포인트

### BuyingPowerCalculator

```java
public class BuyingPowerCalculator {
    public BigDecimal calculate(Account account) {
        BigDecimal cash = account.getCashBalance();
        
        // 보유 주식의 현재 시가 기준 대용가 계산
        BigDecimal collateral = BigDecimal.ZERO;
        for (Holding holding : account.getHoldings()) {
            BigDecimal currentPrice = getRealtimePrice(holding.getStockCode());
            BigDecimal value = currentPrice
                .multiply(new BigDecimal(holding.getQuantity()))
                .multiply(new BigDecimal("0.7")); // 대용가율 70%
            collateral = collateral.add(value);
        }
        
        BigDecimal pendingSales = account.getPendingSalesAmount();
        BigDecimal totalPower = cash.add(collateral).add(pendingSales);
        
        return totalPower.divide(
            new BigDecimal("0.4"), 
            2, 
            RoundingMode.DOWN
        );
    }
    
    // 실시간 가격 조회
    private BigDecimal getRealtimePrice(String stockCode) {
        // Redis 캐시에서 조회 또는 외부 API 호출
        return priceService.getCurrentPrice(stockCode);
    }
}
```

### LiquidationEngine

```java
public class LiquidationEngine {
    public void liquidate(Account account) {
        BigDecimal debtAmount = account.getDebtAmount();
        
        for (Holding holding : account.getHoldings()) {
            // 전일 종가 기준 하한가 계산
            BigDecimal prevClose = getPreviousClosePrice(holding.getStockCode());
            BigDecimal lowerLimit = prevClose.multiply(new BigDecimal("0.9"));
            
            // 안전 마진을 고려한 청산 수량
            int quantity = debtAmount
                .divide(lowerLimit, 0, RoundingMode.UP)
                .intValue();
            
            // 보유 수량 초과하면 전량 매도
            quantity = Math.min(quantity, holding.getQuantity());
            
            createForcedSellOrder(account, holding, quantity);
            
            debtAmount = debtAmount.subtract(
                lowerLimit.multiply(new BigDecimal(quantity))
            );
            
            if (debtAmount.compareTo(BigDecimal.ZERO) <= 0) {
                break;
            }
        }
    }
}
```

## 백엔드 챌린지

1. **실시간 가격 반영**: 매수 가능 금액 계산 시 현재가 조회
2. **금액 정합성**: BigDecimal로 1원 단위까지 정확
3. **가격 변동 대응**: 청산 수량 계산 시 안전 마진 확보
4. **체결 불확실성**: 시장가 주문 시 예상가와 실제 체결가 차이
5. **동시성**: 여러 계좌 동시 처리 시 가격 데이터 일관성

가격이 계속 움직이는 상황에서 금액 하나 안 틀리게 만드는 게 핵심.
