# 인증·로그인 정리

JWT, 로그인 방식, SSO/OAuth2 위임 처리에 대한 정리 문서다.

---

## 1. JWT를 왜 로그인할 때 쓰냐

- **로그인** = 비밀번호 등으로 “이 사람이 누구인지” 한 번 확인하는 단계.
- 그 다음부터는 매 요청마다 비밀번호를 다시 보낼 수 없으므로, “이미 인증된 사용자”라는 걸 **어떤 값**으로 나타내서 보내야 함.
- 그 “값”을 **토큰**으로 주는 방식 중 하나가 JWT.

**정리**: 로그인 **성공 시점에 JWT를 발급**하고, 이후 요청에서는 그 JWT만 보내면 서버가 비밀번호 없이 “누구인지” 검증할 수 있음. 즉, **“로그인 결과를 유지하는 수단”**으로 JWT를 쓰는 것.

---

## 2. 굳이 JWT여야 하냐

아니요. 대안이 있다.

| 방식 | 동작 | 특징 |
|------|------|------|
| **세션** | 로그인 시 서버가 세션 저장소(Redis, DB)에 사용자 정보 저장 → 세션 ID만 쿠키로 전달. 이후 요청마다 세션 ID로 저장소 조회. | 로그아웃·강제 만료가 쉬움. 서버에 상태 필요. |
| **JWT** | 로그인 시 서명된 토큰 발급. 클라이언트가 요청마다 토큰 전달. 서버는 저장소 조회 없이 서명만 검증. | Stateless. 확장·공유 편함. **토큰 폐기**(로그아웃, 비밀번호 변경 시 무효화)가 까다로움. |
| **Opaque Token** | 랜덤 문자열 토큰 발급 + DB/캐시에 “토큰 → 사용자” 매핑 저장. 요청마다 토큰으로 조회. | JWT처럼 토큰을 쓰지만, 검증 시마다 저장소 조회. 폐기는 세션처럼 처리 가능. |

---

## 3. 로그아웃·탈취 시 무효화

JWT는 서버가 “발급한 토큰 목록”을 안 갖고 있어서, **즉시 무효화**가 기본 설계에 없음.

**실무**: **기기 ID(또는 세션 ID)와 토큰(또는 token id)을 함께 저장**해 둠.

- **로그아웃** → 해당 기기/세션에 매핑된 토큰 레코드 삭제 또는 revoked 처리. 그 토큰으로 오는 요청은 401.
- **탈취 대응** → “이 토큰(또는 이 기기/세션) 무효”라고 저장소에 반영하면, 탈취자가 가진 토큰은 더 이상 사용 불가.

즉, **로그아웃이든 탈취 대응이든**, “기기/세션과 매칭해 둔 토큰을 서버 저장소에서 지우거나 revoked 처리”하면 됨.

---

## 4. “탈취됐다”고 어떻게 판단하냐

**자동**
- **기기/지역 불일치**: 토큰 발급 시 저장한 deviceId/fingerprint/IP와 현재 요청의 기기·IP가 다름.
- **Impossible travel**: 짧은 시간에 물리적으로 불가능한 지역에서 동일 토큰 사용.
- **동시 사용·다지역**: 같은 사용자 토큰이 동시에 다른 지역에서 사용됨.
- **이상 행동**: 평소와 다른 API 호출 패턴.

**수동**
- 사용자: “제가 로그인한 적 없는 기기에서 접속된 것 같아요”, “비밀번호 바꿀게요”.
- 운영: 신고 접수 시 해당 계정 세션/토큰 무효화.

---

## 5. 탈취 전에 이미 유출됐을 수 있음

**이미 그 JWT로 API 호출해서 개인정보를 읽고 나간 뒤**에, 나중에 “탈취됐다” 판단하고 토큰 무효화해도 **그때까지의 유출은 되돌릴 수 없음**. 무효화는 “그 다음부터 못 쓰게” 하는 **추가 피해 방지**일 뿐.

**대응**
- **예방**: 최소 권한, 토큰 짧은 수명, HTTPS, 클라이언트 보안 저장.
- **사후**: **감사 로그**로 “이 토큰으로 언제·어떤 API·어떤 리소스” 접근했는지 기록 → 유출 범위 파악·대응. 빠른 무효화로 추가 사용 차단.

---

## 6. JWT claim은 누구나 읽을 수 있음

JWT는 **header.payload.signature** 구조.

- **header**와 **payload(claim)** 는 **Base64URL 인코딩**일 뿐, **암호화가 아님**.
- **secret 없이** 디코딩만 하면 내용을 다 읽을 수 있음 (예: jwt.io).
- **secret**이 필요한 건 **서명 검증/생성**일 때뿐.

**따라서** claim에는 비밀번호·민감 정보를 넣지 말고, 식별자(userId 등)·권한 정도만 넣는 게 맞음.

---

## 7. userId도 민감정보일 수 있음

맥락에 따라 **민감정보로 볼 수 있음**. (다른 정보와 결합 시 간접 식별 가능, 열거 공격 등.)

그래도 claim에 넣는 이유는 **“누구인지” 판별하는 데 필요한 최소 식별자**이기 때문. 그 대신 **다른 민감 정보는 claim에 넣지 말자**는 원칙.

**“흔해서” 문제**: JWT claim에 userId 넣는 게 “맞아서”가 아니라, 잘못되거나 불완전한 정보로 코딩한 사람이 많아서 흔해진 것. “많이 하니까 맞다”가 아니라 “많이 해서 위험이 널리 퍼진다”고 보는 시각이 있음.

---

## 8. JWT는 어디에 적합한가

- **MSA 내부 통신**: 서비스끼리만 토큰 주고받을 때는 JWT가 적합. 클라이언트(브라우저/앱)가 관여하지 않고, 수명 짧게 두기 쉬움.
- **클라이언트에 JWT 넘겨서 “로그인된 사용자 증명”**: 탈취·재전송·payload 노출 가능. “글쎄”라는 시각이 타당함. 세션·불투명 토큰이 더 적합할 수 있음.

---

## 9. “완벽한” 로그인은 없음

클라이언트가 **뭔가 하나** 들고 있어야 “이 사용자다”라고 증명 가능 → 그 **뭔가**를 누군가 가져가면 그 사람이 그 사용자 행세 가능 → **탈취를 원천 차단하는 완벽한 로그인**은 사실상 없음.

**할 수 있는 것**: 탈취를 **어렵게**(HttpOnly, Secure, 짧은 수명, 기기 매핑), 탈취돼도 **피해를 줄이게**(즉시 무효화, 최소 권한, 감사 로그, 이상 탐지). **완벽 대신 완화·대응**을 설계하는 것.

---

## 10. 클라이언트 요청을 100% 믿을 수 있냐

아니요.

- **재전송(Replay)**: 정당한 토큰을 가로채서 다시 보낼 수 있음.
- **탈취**: 토큰이 유출됐다면, 그 토큰을 가진 사람이 요청하는 것. 서버는 “토큰 유효함”만 알 수 있음.
- **요청 내용**: 토큰은 “누구”만 담고, 요청 본문/경로/파라미터는 별도 검증·권한 체크 필요.

토큰은 **“신원 + 무결성”** 검증용으로만 쓰고, 요청 내용은 검증·권한·비즈니스 로직으로 다시 확인해야 함.

---

## 11. 멘토가 JWT를 언급한 이유 (추측)

- **“왜 JWT냐?”**부터 스스로 설명하게 하려고.
- JWT의 **한계**(탈취·폐기·payload 노출)를 스스로 정리하게 하려고.
- **“흔한 선택 = 올바른 선택”**이 아니라는 걸 인지하게 하려고.
- 설계를 **“선택과 트레이드오프”**로 보게 하려고.

즉, JWT가 틀렸다는 지적이 아니라 **“그 선택을 당연시하지 말고, 이유와 한계를 말할 수 있게 하려고”**였을 가능성이 큼.

---

## 12. 대기업·큰 서비스의 로그인 시스템

| 구분 | 내용 |
|------|------|
| **전담 인증 서비스** | 로그인을 한 곳에서만 처리. 나머지 서비스는 그 결과만 신뢰. |
| **웹** | 세션 + 쿠키 (Redis/DB). HttpOnly, Secure, SameSite. 로그아웃 = 세션 삭제. |
| **앱/SPA** | JWT(짧은 수명) + Refresh Token(DB/Redis에 저장·폐기 가능). |
| **직원/내부** | SSO(OAuth2, SAML)로 Okta, Azure AD 등에 위임. |
| **레이어** | MFA, 기기/세션 관리, 이상 탐지, 감사 로그. |

---

## 13. SSO / OAuth2 위임받은 쪽 처리

**위임받은 쪽(우리 앱)** 은 IdP가 인증한 결과를 **한 번만** 쓰고, 그걸로 **우리만의 로그인 상태**를 만듦.

| 단계 | 하는 일 |
|------|---------|
| IdP redirect | **authorization code** 받음 |
| code 교환 | IdP에게 code 보내고 **id_token**(JWT) 등 받음 |
| 검증 | id_token **서명**을 IdP **공개키(JWKS)** 로 검증 |
| 사용자 매핑 | claim(sub, email)으로 **우리 사용자** 찾기/만들기 |
| 우리 로그인 | **우리 세션** 또는 **우리 토큰** 발급 → 클라이언트에 전달 |
| 이후 | 모든 요청은 **우리 세션/토큰**으로만 처리. IdP 토큰은 로그인 이벤트 때 한 번만 사용 |

---

## 14. 로그인에 쓸 수 있는 것 (요약)

- **웹, 로그아웃/무효화가 중요하면**: **세션(쿠키 + 서버 저장소)** 또는 **불투명 토큰 + 저장소**.
- **모바일/API, JWT를 쓰고 싶으면**: **JWT(짧은 수명) + Refresh Token(서버 저장·폐기)**.
- **클라이언트에 넘겨서 유저 증명** 용도로는 **세션/불투명 토큰**이 JWT보다 더 적합하다고 보는 시각이 많음.
