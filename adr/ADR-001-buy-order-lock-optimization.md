# ADR-001: 매수 주문 비관적 락 최적화 전략

## 문제

매수 주문 접수 시 Account row에 `PESSIMISTIC_WRITE` 락을 잡고 있는 동안:
- 미결제 합계 조회 (Supplier)
- PENDING 주문 목록 전체 로드 + Java 스트림 합산

두 가지 추가 쿼리가 락 보유 중에 실행된다.
동시 매수 주문이 증가하면 락 대기 중인 트랜잭션이 DB 커넥션을 점유한 채 대기 → 커넥션 풀 고갈 위험.

## 대안

### 1. DB SUM 쿼리로 교체
PENDING 주문 목록 전체를 로드해 Java에서 합산하는 대신,
`SELECT SUM(reserved_margin) ...` 한 번으로 교체.
락 보유 중 처리 시간 단축.

### 2. Account에 예약증거금 합계 컬럼 유지
Account row에 `reservedMarginSum` 컬럼을 두고 주문 접수/취소마다 갱신.
락 안에서 추가 쿼리 없음.

### 3. Redis 캐시 활용
예약증거금 합계를 Redis에 저장, 락 안에서 Redis 읽기만 수행.
DB 쿼리 없음. 단 Redis ↔ DB 동기화 필요.

### 4. Kafka accountId 파티셔닝으로 직렬화 (Phase 3)
같은 계좌 주문을 항상 같은 Consumer 스레드에서 순차 처리.
DB 비관적 락 자체 제거 가능.

## 트레이드오프

| 대안 | 장점 | 단점 |
|---|---|---|
| SUM 쿼리 | 변경 범위 작음, 즉각 효과 | 락 구조 자체는 유지 |
| Account 컬럼 | 락 안 추가 쿼리 없음 | 쓰기 부하 증가, 정합성 관리 포인트 추가 |
| Redis 캐시 | 락 안 DB 쿼리 없음 | Redis-DB 원자성 보장 불가, 잔고 계산 오차 위험 |
| Kafka 직렬화 | 락 완전 제거, 확장성 | 레이어 추가, HTTP 응답 방식 변경, 복잡도 증가 |

## 결정

- **단기 (현재)**: 대안 1 — DB SUM 쿼리로 교체. 변경 범위 최소, 리스크 없음.
- **장기 (Phase 3)**: 대안 4 — Kafka accountId 파티셔닝. SUM 쿼리도 불필요해짐.

Redis 캐시(대안 3)는 금융 도메인에서 잔고 계산 오차가 허용되지 않아 제외.
Account 컬럼(대안 2)은 정합성 관리 복잡도 대비 효과가 크지 않아 제외.

## 결과

- 락 보유 중 쿼리 N개 → 1개로 감소 (SUM 쿼리)
- Phase 3에서 Kafka 직렬화 도입 시 비관적 락 전면 제거 예정
- HTTP 응답이 `202 + requestId`로 바뀌고 클라이언트는 폴링/WebSocket으로 결과 수신
